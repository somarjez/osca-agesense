## 1. How Model Distribution Works

The system uses a design called **notebook overrides** to guarantee that every machine produces identical results:

1. You train the model in `osca5.ipynb` and export the results to `osca_output/`
2. You copy those files into `python/models/` inside the repository
3. You push to GitHub
4. Every other machine pulls and reseeds — they read the validated CSV files instead of running live ML inference

This means the model does **not** run differently on different machines. Every device uses your exact notebook output.

The key setting that enables this is:

```env
ENABLE_NOTEBOOK_OVERRIDES=true
```

When this is `true` (the default), the inference service reads `composite_risk`, `cluster_id`, and `risk_level` from `python/models/predictions/senior_predictions.csv` — the file you committed — instead of computing them live.

## 2. Machine Roles at a Glance

| Role | Machine | Has notebook? | Has `osca_output/`? | Action when model changes |
|---|---|---|---|---|
| **Training machine** | The developer who retrains | Yes | Yes (generated by notebook) | Follow Section 4 below |
| **Other developer machines** | Everyone else | Not needed | Not needed | Follow Section 5 below |

> No other machine needs the notebook, Python training environment, or `osca_output/` folder. They only need what is in the git repository.

## 3. What Files Need to Change When You Retrain

When you retrain the notebook, these are the files that change and must be updated in the repository:

### Model files (`python/models/`)

These are the trained `.pkl` and `.json` artefacts the system loads at startup:

| File | Changes when... |
|---|---|
| `scaler.pkl` | You retrain on new data or change features |
| `umap_nd.pkl` | You retrain UMAP |
| `kmeans.pkl` | You retrain KMeans |
| `edu_encoder.pkl` | You change education encoding |
| `income_encoder.pkl` | You change income encoding |
| `gbr_*.pkl`, `rfr_*.pkl` | You retrain the risk regressors |
| `feature_list.json` | You change which features go into clustering |
| `ml_risk_features.json` | You change which features go into risk scoring |
| `cluster_mapping.json` | You change how raw KMeans IDs map to group names |
| `cluster_metadata.json` | You update cluster names or descriptions |
| `vif_retained_features.json` | You change the VIF filtering step |
| `asset_weights.json` | You update scoring weights |

### Prediction files (`python/models/predictions/`)

These are the per-senior validated results from the notebook — the source of truth for the dashboard:

| File | Changes when... |
|---|---|
| `senior_predictions.csv` | Any retrain — contains composite_risk, cluster_id, risk_level per senior |
| `senior_recommendations_flat.csv` | Any retrain — contains recommended actions per senior |

> **These two CSV files are the most important.** Even if you only rerun the notebook on the same data (without changing the model architecture), these files may have updated values and must be pushed.

## 4. Training Machine — Full Update Workflow

Do these steps in order every time you retrain the notebook.

---

### Step 1 — Retrain the notebook

Run `osca5.ipynb` from top to bottom in Jupyter. Verify that `osca_output/` is generated with:

```
osca_output/
├── model/
│   ├── *.pkl         (scaler, umap, kmeans, encoders, regressors)
│   └── *.json        (feature lists, cluster mapping, weights)
└── predictions/
    ├── senior_predictions.csv
    └── senior_recommendations_flat.csv
```

Do not proceed until the notebook runs without errors.

---

### Step 2 — Sync files into the repository

**Option A — Use `setup.bat` (recommended):**

Run `setup.bat`. Step 11 does the sync automatically:

```
Double-click setup.bat
```

It copies all `.pkl` and `.json` files from `osca_output/model/` and both CSVs from `osca_output/predictions/` into the correct locations inside `python/models/`.

**Option B — Manual copy (if you want to run it yourself):**

Open Command Prompt in the project root and run:

```bat
xcopy /Y ..\osca_output\model\*.pkl  python\models\
xcopy /Y ..\osca_output\model\*.json python\models\
xcopy /Y ..\osca_output\predictions\senior_predictions.csv          python\models\predictions\
xcopy /Y ..\osca_output\predictions\senior_recommendations_flat.csv python\models\predictions\
```

> `osca_output/` is located one level above the project root (at `../osca_output/` relative to the project).

---

### Step 3 — Validate

Run the two validation scripts to confirm the new files are correct before pushing:

```bat
python\venv\Scripts\python python\tests\test_inference_paths.py
python\venv\Scripts\python python\tests\test_inference_e2e.py
```

Expected output: all tests pass. If any test fails, do not push — fix the issue first.

What these tests check:
- `test_inference_paths.py` — model files exist, CSV loads, urgency logic, priority flag thresholds
- `test_inference_e2e.py` — end-to-end infer() on known seniors (Norlito Basa urgent, Rosa Amante moderate), checks composite within 0.001 of CSV value, notebook_override_applied=True

---

### Step 4 — Restart Flask services and verify the dashboard

Before committing, do a quick sanity check:

1. Start the system: `start.bat`
2. Go to `/ml/status` — confirm both services show `ok`
3. Go to the dashboard — confirm the risk distribution matches your notebook output:
   - HIGH: 53, MODERATE: 186, LOW: 36, Urgent: 1

If the numbers are wrong, check that `ENABLE_NOTEBOOK_OVERRIDES=true` is in your `.env` and that the Flask services reloaded the new CSV (restart them if needed via `/ml/status`).

---

### Step 5 — Commit and push

```bash
git add python/models/
git commit -m "model: update trained files from notebook rerun YYYY-MM-DD"
git push
```

Replace `YYYY-MM-DD` with today's date (e.g., `model: update trained files from notebook rerun 2026-05-13`).

> Only add `python/models/` — do not accidentally commit `osca.csv`, `.env`, `osca_output/`, or `python/venv/`.

## 5. Other Machines — What They Need to Do

After you push, any other machine (collaborator, second laptop, presentation machine) needs to do the following to get the updated model:

---

### Step 1 — Pull the latest changes

```bash
git pull
```

This downloads the new model files and prediction CSVs that were just committed.

---

### Step 2 — Run `start.bat` once before reseeding

```
Double-click start.bat
```

This ensures any new `.env` keys added to `.env.example` (such as `ENABLE_NOTEBOOK_OVERRIDES=true`) are automatically appended to the local `.env` file.

> **Do not skip this step.** If you reseed before running `start.bat`, the seeder may run with outdated `.env` values and produce wrong results.

You can close `start.bat` immediately after it starts — you do not need to keep the server running.

---

### Step 3 — Reseed the database

```bash
php artisan migrate:fresh --seed
```

This:
1. Drops all tables
2. Re-runs all migrations
3. Re-imports senior data from `osca.csv`
4. Runs the full ML pipeline for each senior using the new model files
5. Saves results (cluster, risk level, composite score, recommendations) to the database

Expected output at the end: `ML success: 275, fallback: 0, errors: 0`

Expected time: 10–20 minutes (the ML pipeline runs for all 275 seniors).

---

### Step 4 — Verify the dashboard

Start the system with `start.bat` and open the browser. The dashboard should show:

| Metric | Expected value |
|---|---|
| Total seniors | 275 |
| HIGH risk | 53 |
| MODERATE risk | 186 |
| LOW risk | 36 |
| Urgent | 1 (Norlito M. Basa) |
| Pending recommendations | ~2,114 |

If the numbers differ, see [Troubleshooting](#troubleshooting-wrong-dashboard-numbers) below.

## 7. Common Mistakes

### Wrong dashboard numbers after reseeding

**Symptom:** HIGH=32 or similar wrong distribution after `migrate:fresh --seed`.

**Cause:** `ENABLE_NOTEBOOK_OVERRIDES` is missing or set to `false` in `.env`. The seeder always runs before Flask starts, so it uses the local subprocess. If the flag is `false`, the subprocess runs live ML inference instead of reading the validated CSV.

**Fix:**
1. Open `.env` in the project root
2. Find `ENABLE_NOTEBOOK_OVERRIDES` and set it to `true`
3. If the key is missing entirely, run `start.bat` — it will add it automatically from `.env.example`
4. Re-run `php artisan migrate:fresh --seed`

---

### Correct numbers after seeding, but wrong after single-senior analysis

**Symptom:** Batch results are correct but running analysis on one new senior shows wrong cluster.

**Cause:** Flask is not running, or loaded stale model files from before your push.

**Fix:**
1. Go to `/ml/status` — confirm both services show `ok`
2. If they show `unreachable`, click **Start ML Services** and wait 30–60 seconds
3. If they were already running before your `git pull`, restart them — they cache model files at startup

---

### `osca.csv not found` during seeding

The seeder looks for `osca.csv` in the project root first, then one level above.

Place the file here:

```
osca-agesense/
├── osca.csv          ← project root (recommended)
├── setup.bat
├── composer.json
```

---

### Flask services do not pick up new model files

The Flask services load model files at startup using `lru_cache` — they do not reload while running. After a `git pull` that includes new model files, restart the services:

1. Go to `/ml/status`
2. Click **Stop ML Services** (or close the terminal running the services)
3. Click **Start ML Services**
4. Wait for both services to show `ok`

---

### Validation tests fail after syncing new model files

If `test_inference_e2e.py` fails with a composite delta > 0.001, the prediction CSV and the model `.pkl` files are out of sync — they are from different notebook runs.

**Fix:** Re-run the notebook from top to bottom and re-sync all files together in a single step (do not mix model files from one run with CSVs from another).
